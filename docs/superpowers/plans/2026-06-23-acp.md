## Why / what
ACP is how SeldonFrame's paid offerings get bought **inside ChatGPT** (OpenAI Instant Checkout), card-based. It complements the ChatGPT App (which builds workspaces + installs FREE agents): ACP completes the **paid** purchase of a marketplace agent in-conversation. Two parts of the protocol:
1. **Product feed** â€” a catalog ChatGPT ingests. We emit our published marketplace agents.
2. **Agentic Checkout** â€” REST endpoints ChatGPT calls: create â†’ update â†’ complete a checkout session, with **delegated payment** (a Shared Payment Token the merchant charges via its own Stripe). We implement the shapes; the charge sits behind a pluggable processor that is **stubbed (no money) in v1**.

**Money-safety (airtight, non-negotiable):**
- `AcpPaymentProcessor` interface; the only impl wired by default is `devStubProcessor` â†’ returns a simulated `paymentRef` and charges NOTHING.
- A real `stripeDelegatedProcessor` is written but throws unless `process.env.ACP_LIVE === "true"` AND Stripe delegated-payment creds are present. The `complete` handler refuses any non-stub processor unless `ACP_LIVE==="true"`. Default deploy = stub = zero charges.
- The SF 5% marketplace fee is **computed + recorded** on the (simulated) order via `computeMarketplaceFeeCents` (reuse `lib/billing/gmv.ts`) â€” not charged in v1.

**Reuse (verified earlier):** marketplace listing query `listMarketplaceAgentsFromDb()` + `MarketplaceAgentRow` (`lib/marketplace/agent-listings.ts`); fee primitive `computeMarketplaceFeeCents` (`lib/billing/gmv.ts`); the x402 pluggable-verifier pattern to mirror (`lib/marketplace/x402.ts` â€” read it for the stub/flag idiom); stateless HMAC pattern (`lib/marketplace/rental-token.ts`) for signing the feed/session integrity if needed. Stripe client lives at `lib/payments/providers/stripe.ts` â€” read but DO NOT call a charge in v1.

## ACP shapes (the target â€” implement these exactly)
`CheckoutSession`: `{ id, status: "not_ready_for_payment"|"ready_for_payment"|"completed"|"canceled", currency, line_items:[{ id, item:{ id, quantity }, base_amount, subtotal, tax, total }], totals:[{ type:"subtotal"|"tax"|"total", display_text, amount }], buyer?:{ first_name,last_name,email,phone_number }, payment_provider:{ provider:"stripe", supported_payment_methods:["card"] }, messages:[{ type:"info"|"error", text }], order?:{ id, checkout_session_id, permalink_url } }`. Amounts are integer minor units (cents).
- `POST /api/acp/checkout_sessions` (create): body `{ items:[{ id, quantity }], buyer? }` â†’ 201 CheckoutSession (status `ready_for_payment` once items resolve).
- `GET /api/acp/checkout_sessions/{id}` â†’ CheckoutSession.
- `POST /api/acp/checkout_sessions/{id}` (update): body partial `{ items?, buyer? }` â†’ CheckoutSession.
- `POST /api/acp/checkout_sessions/{id}/complete`: body `{ buyer?, payment_data:{ token, provider:"stripe" } }` â†’ runs the processor â†’ on stub-ok returns CheckoutSession `status:"completed"` + `order`. **On the free/zero path or stub, no charge.**
- `POST /api/acp/checkout_sessions/{id}/cancel` â†’ status `canceled`.
- Errors: `{ type:"invalid_request"|"processing_error", code, message, param? }` + correct HTTP status. Header `Idempotency-Key` honored on create/complete (dedupe). A request signature header is verified ONLY when `ACP_WEBHOOK_SECRET` is set (flagged; absent in v1).

`item.id` = a marketplace agent `slug` (the feed uses slug as the product id).
