## Why / what
AP2 is Google's open **Agent Payments Protocol**: a payment-method-agnostic **mandate** layer that proves user intent with signed credentials, and it **natively supports x402** as a settlement rail. SeldonFrame already has the x402 rail merged (inert). AP2 v1 adds the **mandate verification + cart authorization layer on top**, settling through the existing x402 path — which opens SF to Google/Gemini + the UCP umbrella while reusing everything already built. It's thin: mostly pure mandate logic + a bridge to x402.

The three AP2 mandates (signed artifacts — "VDCs"):
- **IntentMandate** — a user grants an agent authority to transact within constraints (`max_amount`, `currency`, `merchants?`, `expiry`).
- **CartMandate** — the specific cart the user/agent approves (`items`, `total`, `currency`, `merchant`, optional `intent_ref`, `expiry`).
- **PaymentMandate** — authorizes payment for a cart via a method (`cart_ref`, `payment_method:{ type:"x402"|"card" }`, `amount`, `currency`).

**Money-safety:** AP2 only *verifies* mandates. Actual settlement is delegated to the **existing x402 `SettlementVerifier`**, whose only wired impl is `devStubVerifier` → **no USDC moves**. AP2 adds zero new money path. The `vdcVerifier` (real W3C-VC/DID signature verification) is a documented stub; v1 uses a real but simple **HMAC detached signature** over canonical JSON (verifiable, table-free, mirrors `rental-token.ts`).

**Reuse — READ `packages/crm/src/lib/marketplace/x402.ts` FIRST** for exact exports: the payment-requirements builder, the `X-PAYMENT` header parser, the `SettlementVerifier` interface + `devStubVerifier`, `USDC_DECIMALS`, `BASE_USDC_ASSET`, and the house-org/pay-to/facilitator env gating. AP2's bridge composes these — do not duplicate them. Also reuse the HMAC idiom in `lib/marketplace/rental-token.ts` and `seldonframe_events` logging.

**Tech:** Next.js 16, `node --import tsx --test`, TDD pure logic. tsc 0-new; `check-use-server` clean. **No migration.**

## Task 1: Pure mandate types + signing/verification (TDD)
**Files:** Create `packages/crm/src/lib/ap2/mandates.ts` + test `packages/crm/tests/unit/ap2/mandates.spec.ts`.
- Types: `IntentMandate`, `CartMandate`, `PaymentMandate` (each `{ ...payload, signature: string }`), plus `MandateVerdict = { kind:"valid" } | { kind:"invalid"|"expired"|"constraint_violation", reason }`.
- `canonicalize(payload)` — stable JSON (sorted keys) for signing. `signMandate(payload, secret)` / `verifyMandateSignature(mandate, secret)` — HMAC-SHA256 detached signature, constant-time compare (reuse the pattern from `rental-token.ts`; inject `secret` + `now`).
- `MandateVerifier` interface `{ verify(mandate, ctx): MandateVerdict }`; `devStubVerifier` = HMAC + expiry check (real); `vdcVerifier` = throw-only documented stub ("AP2 VDC verification not configured" — real W3C-VC/DID path, not implemented).
- Pure checks: `verifyIntentConstraints(intent, cart)` (cart.total ≤ intent.max_amount, currency match, merchant allowed, intent not expired); `verifyCartMatchesPayment(cart, payment)` (amount + currency + cart_ref agree). All take an injected `now` — **no Date.now**.
